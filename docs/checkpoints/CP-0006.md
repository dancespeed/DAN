# CHECKPOINT — DAN Platform

## Текущий этап

Переход от временной событийной архитектуры к постоянной системе сообщений:

```text
Event
  ↓
EventBus
  ↓
EventDispatcher
  ↓
Adapter
  ↓
MessageBus
  ↓
MessageDispatcher
```

Существующий контур `EventBus → EventDispatcher` пока сохраняется и продолжает работать.

---

## Принятые архитектурные решения

### 1. Реестры протокола вынесены отдельно

Созданы файлы:

```text
message/module_id.hpp
message/message_id.hpp
```

Они относятся к постоянному протоколу DAN и не размещаются в `config`.

### 2. Сохраняется единый стиль проекта

Используются:

```cpp
#include <stdint.h>
```

и глобальные типы:

```cpp
uint8_t
uint16_t
```

без пространства имён `std`.

### 3. `ModuleId` — постоянный реестр модулей

`module_id.hpp` содержит:

```cpp
using ModuleId = uint8_t;
```

и именованные идентификаторы в пространстве имён:

```cpp
Module::None
Module::System
Module::Button
Module::PWM
Module::EEPROM
Module::CAN
Module::Logic
Module::Sequencer
Module::Diagnostic
Module::Broadcast
```

Значения `ModuleId` задаются явно.

Особо зафиксировано:

```text
EventTarget::Broadcast = 0
Module::Broadcast      = 0xFF
```

Поэтому прямое преобразование `EventTarget → ModuleId` недопустимо.

### 4. `MessageId` пока зависит от `EventType`

На этапе миграции `EventType` остаётся источником числовых значений:

```cpp
constexpr MessageId IgnitionOn =
    static_cast<MessageId>(EventType::IgnitionOn);
```

Это позволяет:

* не создавать второй независимый числовой реестр;
* исключить рассинхронизацию;
* сохранить простую миграцию;
* отложить фиксацию окончательных номеров протокола до удаления `Event`.

Из-за существующего типа `Message` используется пространство имён:

```cpp
MessageIds
```

а не `Message`.

## Проверка

После исправления:

```cpp
#include <stdint.h>

using ModuleId  = uint8_t;
using MessageId = uint16_t;
```

проект успешно собрался без ошибок.

Поведение системы не изменялось.

## Следующий кирпичик

Спроектировать `Adapter`.

Предварительно согласованная ответственность:

```cpp
bool Publish(ModuleId sender, const Event& event);
```

`Adapter` должен только:

1. принять отправителя и `Event`;
2. преобразовать `EventTarget` в `ModuleId`;
3. сформировать `Message`;
4. заполнить заголовок и payload;
5. передать сообщение в `MessageBus`.

`Adapter` не должен содержать:

* бизнес-логику;
* маршрутизацию;
* обработку сообщений;
* управление модулями;
* собственный реестр идентификаторов.

## Точка продолжения

Следующая работа начинается с обсуждения внутренней архитектуры `Adapter`. Код пишется только после её утверждения.
