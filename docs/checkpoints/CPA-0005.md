# CPA-0001.md

# DAN Platform
## Architect Checkpoint
## CPA-0005

Назначение

Документ предназначен исключительно для восстановления архитектурного контекста.

---

# Где остановились

Построен фундамент новой платформы сообщений.

Message

↓

MessageQueue

↓

MessageBus

↓

MessageDispatcher

Все компоненты компилируются.

Ни один из них пока не подключен к рабочей системе.

Это сделано намеренно.

---

# Почему принято такое решение

Изначально планировалось переписывать main.cpp.

После анализа реального main.cpp было принято другое решение.

Выяснилось, что main выполняет три независимые обязанности:

1. инициализация

2. генерация Heartbeat

3. диспетчеризация Event

Третья обязанность является отдельной архитектурной ответственностью.

Поэтому сначала принято решение создать Dispatcher.

---

# Самое важное архитектурное решение

Dispatcher — постоянная часть DAN.

Он НЕ является временным переходником.

Переходником будет Adapter.

Dispatcher останется в системе навсегда.

---

# Миграционная стратегия

Не переписывать систему.

Вытеснять старую архитектуру новой.

Последовательность:

Message

↓

MessageQueue

↓

MessageBus

↓

MessageDispatcher

↓

EventDispatcher

↓

Adapter

↓

Input

↓

Logic

↓

PWM

↓

Удаление EventBus

---

# Почему выбран именно такой путь

Главный принцип

Каждый кирпичик должен оставлять систему полностью работоспособной.

Никаких больших переписываний.

Никаких смешанных состояний.

Никаких изменений сразу в нескольких подсистемах.

---

# Очень важное правило

Не смешивать Event и Message внутри Logic.

Не смешивать Event и Message внутри PWM.

Не смешивать Event и Message внутри Input.

Любое преобразование выполняется только в Adapter.

Это архитектурный инвариант.

---

# Что необходимо помнить

MessageBus уже готов.

MessageDispatcher уже существует.

Он пока пустой специально.

Это нормально.

Он станет местом подключения:

MessageValidator

↓

ObjectRegistry

↓

ObjectValidator

↓

Logic

↓

Services

Именно в таком порядке.

---

# Следующий кирпичик

Создать EventDispatcher.

Не менять поведение.

Только вынести существующую диспетчеризацию из main.cpp.

После этого main станет координатором.

---

# После этого

Будет создан Adapter.

Через него начнется миграция Event → Message.

---

# Особенности работы с пользователем

Пользователь предпочитает:

- обсуждать архитектуру до кода;
- получать полный файл, а не патч;
- самостоятельно переносить изменения;
- двигаться маленькими проверяемыми кирпичиками;
- после каждого кирпичика выполнять сборку;
- после успешной сборки переходить к следующему этапу.

Никогда не перепрыгивать через этапы.

При возникновении более красивого архитектурного решения сначала обсудить его, только потом менять план.

Это обязательное правило проекта.
