## FROZEN DECISIONS

### Protocol independence

The DAN protocol does not depend on device configuration.

Static protocol definitions:

- `ModuleId`
- `ObjectId`
- `MessageId`

Device configuration declares only which modules are enabled on a concrete device.

### Module address space

- `0x00` — reserved and invalid as sender or receiver
- `0x01..0xFE` — real modules
- `0xFF` — broadcast receiver

Rules:

- sender must never be `0x00`
- receiver must never be `0x00`
- broadcast is valid only as receiver
- broadcast is invalid as sender

### ObjectId

`ObjectId` is part of the DAN protocol and does not depend on a concrete device.

An object may be absent on a device, but its protocol identifier never changes.

Object identifiers will use reserved functional ranges. Exact ranges are not yet frozen.

### Message structure

The DAN message remains exactly 8 bytes:

- Header — 32 bits
- ObjectId — 16 bits
- Value — 16 bits

Header fields:

- Sender — 8 bits
- Receiver — 8 bits
- Flags — 4 bits
- MessageId — 12 bits

Flags layout:

- bits `0..1` — `MessageType`
- bits `2..3` — `MessageAttribute`

Accepted naming:

- `MessageFlag` → `MessageAttribute`
- `GetFlags()` → `GetAttributes()`
- `HasFlag()` → `HasAttribute()`
- parameter `flags` → `attributes`

`GetAttributes()` returns only `AttributeMask` bits.

Header creation combines fields independently:

- `type & TypeMask`
- `attributes & AttributeMask`

### Value semantics

The message value remains a raw 16-bit field.

Its meaning is defined by the object descriptor, not encoded in every message.

Initial value types:

- `Bool`
- `UInt16`
- `Int16`

No `Float`, strings, colors, or compound values are included at this stage.

### Runtime object descriptor

The minimal runtime descriptor contains:

- `ObjectId`
- `ValueType`
- `ObjectAccess`
- `Scale`

Initial access modes:

- `ReadOnly`
- `WriteOnly`
- `ReadWrite`

`WriteOnly` is valid for action-like objects such as restart, calibration, save settings, or clear log.

### Scale

Scale is stored as a signed decimal exponent:

`real value = raw value × 10^scale`

Examples:

- `237`, scale `0` → `237`
- `237`, scale `-1` → `23.7`
- `1234`, scale `-2` → `12.34`

The intended storage type is `int8_t`.

### Runtime and protocol metadata separation

The microcontroller stores only execution metadata:

- ObjectId
- ValueType
- Access
- Scale

Human-facing metadata stays outside the firmware:

- name
- description
- unit
- icon
- category
- localization

Units of measurement are not stored in the MCU runtime descriptor.

## ARCHITECTURAL MODEL

Message:
Transfers compact runtime data.

Runtime Descriptor:
Explains how firmware must interpret and validate the data.

Protocol Descriptor:
Provides rich metadata for documentation, tools, UI, and external applications.

Execution and presentation are separate responsibilities.

--------------------------------------------------

## RESOURCE RULE

Descriptors must remain static and lightweight.

Preferred characteristics:

- `constexpr` tables
- Flash storage
- no dynamic allocation
- no strings in runtime descriptors
- no `std::map`
- no virtual hierarchy
